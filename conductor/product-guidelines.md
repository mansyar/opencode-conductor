# Product Guidelines

## Prose Style
**Technical & Concise**: Documentation should use direct, technical language with minimal fluff. Focus on clear, actionable information that helps developers quickly understand and use the plugin.

## Branding Guidelines
**Protocol-Driven**: Emphasize the structured, protocol-based approach in all documentation and messaging. The plugin's identity is built around enforcing development protocols and lifecycle management.

## UX Principles
1. **Zero-Friction Setup**: The plugin should require minimal configuration to get started. Automatic installation and setup should be the default experience.
2. **Clear Feedback**: Provide clear, actionable feedback to users at every step. Error messages should be helpful and guide users toward solutions.
3. **Progressive Disclosure**: Show advanced features only when needed. Keep the initial interface simple and reveal complexity gradually.

## Documentation Standards
- **Command Reference**: Complete reference for all slash commands with examples and usage patterns
- **Workflow Examples**: Step-by-step examples of typical workflows showing the full lifecycle from setup to implementation
- **Troubleshooting Guide**: Common issues and their solutions, organized by symptom

## Writing Guidelines
- Use imperative mood for command documentation (e.g., "Run `/conductor:setup` to initialize")
- Include code examples for all commands and workflows
- Use consistent terminology: "track" for features/bug fixes, "phase" for workflow stages
- Reference the official OpenCode documentation where appropriate

## Visual Guidelines
- Use consistent formatting for command examples: backticks for commands
- Include clear section headers and hierarchical organization
- Use bullet points for lists and step-by-step instructions
- Maintain a clean, readable layout with adequate whitespace
